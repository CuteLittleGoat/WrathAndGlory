0. CEL TEGO ZADANIA
======================================================================

Napraw moduł GeneratorNPC po migracji danych na Firebase.

Obecny stan:
- DataVault loguje się już poprawnie.
- DataVault ładuje dane z Firebase Realtime Database.
- W DataVault dane z /datavault/live są poprawnie rozpakowywane z wrappera firebase-import.json.
- GeneratorNPC przy uruchomieniu wiesza się na etapie „Ładowanie danych...”.
- GeneratorNPC ma już w HTML element bramki logowania, ale przepływ JS nie działa kompletnie.
- W DataVault nadal widoczny jest przycisk „WYLOGUJ / ZABLOKUJ DANE”, który należy usunąć.

Docelowy stan:
1. Użytkownik loguje się tylko raz w dowolnym module:
   - DataVault
   - albo GeneratorNPC
2. Sesja Firebase Auth jest wspólna dla obu modułów na tym samym originie GitHub Pages/PWA.
3. Po zalogowaniu w DataVault wejście do GeneratorNPC nie pyta ponownie o hasło.
4. Po zalogowaniu w GeneratorNPC wejście do DataVault nie pyta ponownie o hasło.
5. Przeglądarka/PWA ma utrzymywać sesję przez Firebase Auth local persistence.
6. Aplikacja nie ma własnoręcznie zapisywać hasła.
7. Przeglądarka może zaproponować zapis hasła dzięki poprawnemu formularzowi password/autocomplete.
8. Aplikacja nie pyta ponownie do czasu utraty sesji, resetu/zmiany danych uwierzytelniających, wyczyszczenia danych strony albo wygaśnięcia/odwołania sesji przez Firebase.
9. Okno logowania w DataVault i GeneratorNPC ma być identyczne wizualnie i tekstowo.
10. Przycisk „WYLOGUJ / ZABLOKUJ DANE” ma zostać usunięty z UI.

WAŻNE:
Nie dodawaj hasła do kodu.
Nie używaj Firestore passwordHash.
Nie wracaj do publicznego data.json.
Nie zmieniaj formatu firebase-import.json.
Nie zmieniaj .gitignore w tym zadaniu.

3. NIE ROBIĆ WŁASNEGO ZAPISU HASŁA
======================================================================

Wymaganie użytkownika brzmi:
„użytkownik loguje się RAZ do któregoś z modułów i aplikacja PWA/przeglądarka trzyma zapisane hasło i nie pyta ponownie do czasu resetu”.

Realizuj to bezpiecznie tak:

1. Nie zapisuj hasła ręcznie w:
   - localStorage,
   - sessionStorage,
   - IndexedDB,
   - cookies,
   - Cache API,
   - plikach JS,
   - Firestore,
   - Realtime Database.

2. Używaj Firebase Auth local persistence:
   browserLocalPersistence

3. Formularz logowania ma mieć:
   input type="password"
   autocomplete="current-password"

4. Przeglądarka/password manager może zaproponować zapis hasła.
   To jest po stronie przeglądarki.

5. Aplikacja ma polegać na zapisanej sesji Firebase Auth.
   Jeśli auth.currentUser istnieje, nie pokazuj bramki hasła.

6. Nie dodawaj przycisku wylogowania w UI.
   Użytkownik resetuje dostęp przez zmianę hasła w Firebase Authentication albo przez wyczyszczenie danych strony/przeglądarki.

Uwaga:
Po resecie/zmianie hasła Firebase może wymusić ponowne uwierzytelnienie zależnie od stanu sesji/tokenów. Kod aplikacji ma poprawnie reagować:
- jeśli sesja działa, ładuje dane,
- jeśli Firebase zwróci brak autoryzacji, pokazuje bramkę hasła.

7. UWAŻAĆ NA PODWÓJNE initializeApp W GENERATORNPC
======================================================================

GeneratorNPC może używać Firebase także do ulubionych przez Firestore.
Nie wolno zepsuć tego mechanizmu, ale trzeba uniknąć konfliktu z DataVault Auth/RTDB.

Problem:
shared/firebase-data-loader.js prawdopodobnie inicjalizuje domyślną aplikację Firebase przez:
initializeApp(window.WG_FIREBASE_CONFIG)

GeneratorNPC może też robić:
initializeApp(window.firebaseConfig)

Jeżeli oba używają domyślnej aplikacji bez nazwy, może pojawić się błąd:
Firebase App named '[DEFAULT]' already exists

Poprawne rozwiązanie:

Opcja A — jeśli favorites używa tego samego projektu Firebase:
- zamiast initializeApp(window.firebaseConfig) użyj getApps()/getApp(),
- użyj istniejącej domyślnej aplikacji.

Opcja B — jeśli favorites ma osobny config/projekt:
- zainicjalizuj aplikację favorites jako nazwaną aplikację:

import { initializeApp, getApps, getApp } from "https://www.gstatic.com/firebasejs/12.6.0/firebase-app.js";

function getOrCreateNamedFirebaseApp(name, config) {
  return getApps().find((app) => app.name === name) || initializeApp(config, name);
}

const favoritesApp = getOrCreateNamedFirebaseApp("generatorNpcFavorites", window.firebaseConfig);
const db = getFirestore(favoritesApp);

Nie wolno inicjalizować drugiego default app, jeśli default app już istnieje.

8. DODAĆ getFirebaseApi() W GENERATORNPC
======================================================================

W głównym module JS GeneratorNPC dodaj helper identyczny logicznie jak w DataVault:

async function getFirebaseApi() {
  if (window.DataVaultFirebaseReady) {
    return window.DataVaultFirebaseReady;
  }

  await new Promise((resolve, reject) => {
    const timeout = setTimeout(() => {
      reject(new Error("FIREBASE_LOADER_NOT_READY"));
    }, 8000);

    window.addEventListener("datavault-firebase-loader-ready", () => {
      clearTimeout(timeout);
      resolve();
    }, { once: true });
  });

  if (!window.DataVaultFirebaseReady) {
    throw new Error("FIREBASE_LOADER_NOT_READY");
  }

  return window.DataVaultFirebaseReady;
}

Używaj tego helpera wszędzie w GeneratorNPC zamiast bezpośrednio:

await window.DataVaultFirebaseReady

10. DODAĆ START PRYWATNEGO PRZEPŁYWU W GENERATORNPC
======================================================================

GeneratorNPC nie może startować od razu od loadData(), jeśli nie ma sesji.
Musi startować od startPrivateDataFlow().

Dodaj:

async function startPrivateDataFlow() {
  let firebaseApi;

  try {
    firebaseApi = await getFirebaseApi();
    await firebaseApi.initFirebaseDataAccess();

    const user = await firebaseApi.waitForAuthReady();

    if (!user) {
      clearGeneratorDataUi();
      setDataStatus(translations[currentLanguage].messages.statusLoadPrivateData || "Ładowanie danych z prywatnej bazy...");
      showAccessGate();
      return;
    }

    hideAccessGate();
    await loadPrivateGeneratorData();
  } catch (error) {
    clearGeneratorDataUi();

    const message = firebaseApi && firebaseApi.getReadableAccessError
      ? firebaseApi.getReadableAccessError(error, currentLanguage)
      : String(error && error.message ? error.message : error);

    setDataStatus(message, true);
    showAccessGate(message);
    console.error("BŁĄD FIREBASE GENERATORNPC:", error);
  }
}

Na końcu głównego modułu GeneratorNPC wywołaj:
startPrivateDataFlow();

Nie wywołuj starego loadData() bezpośrednio przy starcie.

function assertGeneratorDataShape(data) {
  if (!data || typeof data !== "object") {
    throw new Error("GENERATORNPC_DATA_NOT_OBJECT");

  if (!data.sheets || typeof data.sheets !== "object") {
    throw new Error("GENERATORNPC_DATA_MISSING_SHEETS");

  const requiredSheets = [
    "Bestiariusz",
    "Bronie",
    "Pancerze"

  for (const sheet of requiredSheets) {
    if (!data.sheets[sheet]) {
      console.warn(`[GeneratorNPC] Brak oczekiwanego arkusza: ${sheet}`);

  }
}

14. NIE BLOKOWAĆ GENERATORNPC NA „ŁADOWANIE DANYCH...”
======================================================================

Po poprawce nie może istnieć stan, w którym GeneratorNPC bez końca pokazuje:
Ładowanie danych...

W każdym przypadku musi się wydarzyć jedno z poniższych:
1. Brak sesji:
   - pokazuje bramkę hasła,
   - status może mówić „Wymagane logowanie”.
2. Poprawna sesja:
   - ładuje dane,
   - status mówi „OK — załadowano dane z prywatnej bazy”.
3. Błąd hasła:
   - pokazuje „Nieprawidłowe hasło dostępu”.
4. Błąd uprawnień:
   - pokazuje „Brak uprawnień do odczytu prywatnych danych.”
5. Błąd struktury:
   - pokazuje „Dane z Firebase nie mają struktury wymaganej przez GeneratorNPC.”
6. Błąd techniczny:
   - pokazuje czytelny komunikat i loguje szczegóły do console.error.

15. SPÓJNA SESJA MIĘDZY DATAVAULT I GENERATORNPC
======================================================================

Wspólna sesja działa tylko wtedy, gdy:
- oba moduły są na tym samym originie, np. cutelittlegoat.github.io,
- oba używają tego samego shared/firebase-config.js,
- oba używają tego samego shared/firebase-data-loader.js,
- oba używają browserLocalPersistence,
- oba używają Firebase Auth z tym samym projektem i tym samym e-mailem technicznym.

Sprawdź w shared/firebase-data-loader.js, że persistence jest ustawione na:

setPersistence(auth, browserLocalPersistence)

Nie zmieniaj na inMemoryPersistence jako domyślne.
Nie zmieniaj na browserSessionPersistence jako domyślne.

Dopuszczalny fallback:
Jeżeli browserLocalPersistence rzuci błąd w danej przeglądarce, można pokazać czytelny komunikat:
„Przeglądarka blokuje trwałe zapisanie sesji. Sprawdź ustawienia prywatności/cookies/storage dla tej strony.”

Nie ukrywaj tego błędu za ogólnym „Ładowanie danych...”.

======================================================================

Nie naprawiaj już DataVault ładowania danych — ono działa.
