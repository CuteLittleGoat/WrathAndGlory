======================================================================
0. CEL TEGO ZADANIA
======================================================================

Wprowadź poprawki, które CAŁKOWICIE przepinają DataVault i GeneratorNPC z lokalnego/publicznego data.json na Firebase.

Obecny stan po częściowym wdrożeniu:
- generator plików częściowo działa,
- aplikacja potrafi tworzyć data.json oraz firebase-import.json,
- Repozytorium.xlsx jest wybierany lokalnie albo ma być wybierany lokalnie,
- GeneratorNPC ma podmieniony URL na REST Realtime Database,
- ale DataVault nadal zachowuje się tak, jakby źródłem runtime był lokalny/publiczny data.json,
- brakuje pełnej bramki hasła opartej o Firebase Authentication,
- brakuje wspólnego loadera Firebase,
- brakuje rozpakowania wrappera firebase-import.json po stronie runtime,
- zwykły fetch do https://.../datavault/live.json nie wystarczy przy zabezpieczonych regułach.

Docelowy stan:
- DataVault NIE pobiera data.json z repo/hostingu.
- GeneratorNPC NIE pobiera ../DataVault/data.json.
- GeneratorNPC NIE używa gołego fetch do RTDB REST bez Auth.
- Oba moduły korzystają ze wspólnego mechanizmu Firebase Auth + Realtime Database.
- Użytkownik widzi tylko ekran hasła.
- Aplikacja technicznie loguje użytkownika przez Firebase Auth.
- Po zalogowaniu aplikacja pobiera dane z Realtime Database ze ścieżki /datavault/live.
- Aplikacja rozpoznaje wrapper datavault-firebase-import-v1 i wykonuje JSON.parse(dataJson).
- Oba moduły dostają dokładnie taki obiekt, jaki dawniej pochodził z data.json.
- Prawdziwe dane nie są w repo.
- Service worker nie cache’uje prywatnych danych.

Powód:
- aplikacja nadal nie została w pełni przepięta na Firebase jako źródło runtime,
- usunięto plik, którego stara logika próbowała używać jako danych,

- nie ma jeszcze pobierania /datavault/live i rozpakowania dataJson.

To nie jest problem z samym wygenerowaniem firebase-import.json.
To jest problem z tym, że kod aplikacji nadal nie ładuje danych runtime z Firebase w prawidłowy sposób.

======================================================================
2. NIE UŻYWAĆ FIRESTORE passwordHash DO OCHRONY DANYCH
======================================================================

Nie opierać ochrony prywatnego data.json na kolekcji Firestore typu:

admin_security / credentials / passwordHash

Taki mechanizm może służyć najwyżej do ukrywania panelu admina w UI, ale nie do ochrony danych runtime.

Powód:
- sprawdzenie passwordHash w JavaScript jest logiką po stronie klienta,
- użytkownik może obejść logikę UI,
- Realtime Database nie wie, że hasło zostało poprawnie wpisane,
- prawdziwa ochrona danych musi być w regułach Realtime Database,
- reguły RTDB opierają się o auth.uid z Firebase Authentication.

Docelowy mechanizm:
- użytkownik widzi tylko pole „Hasło dostępu”,
- aplikacja technicznie używa Firebase Auth Email/Password:
  signInWithEmailAndPassword(auth, DATA_ACCESS_EMAIL, password)
- Firebase tworzy sesję,
- Realtime Database pozwala czytać /datavault/live tylko wybranemu UID,
- hasło zmienia się w Firebase Authentication, nie w Firestore.

Kolekcję Firestore admin_security/credentials można zostawić dla innych celów, ale nie używać jej jako zabezpieczenia prywatnych danych DataVault/GeneratorNPC.

4. REGUŁY REALTIME DATABASE
======================================================================

Ustaw w Firebase Realtime Database reguły:

{
  "rules": {
    "datavault": {
      "live": {
        ".read": "auth != null && auth.uid === 'raCylxrrJ8YbOtgqXpvH6ygGkn72'",
        ".write": false
      }
    },
    ".read": false,
    ".write": false
  }
}

Znaczenie:
- odczyt /datavault/live tylko po zalogowaniu do Firebase Auth,
- tylko użytkownik z UID raCylxrrJ8YbOtgqXpvH6ygGkn72 ma dostęp,
- zapis z aplikacji webowej jest zabroniony,
- import danych odbywa się ręcznie przez Firebase Console,
- cała reszta bazy zamknięta.

Po ustawieniu reguł niezalogowane wejście na:
https://wh40k-data-slate-default-rtdb.europe-west1.firebasedatabase.app/datavault/live.json

ma zwracać:
{
  "error": "Permission denied"
}

Jeżeli ten adres publicznie pokazuje dane, reguły są błędne.

5. STRUKTURA DANYCH W REALTIME DATABASE
======================================================================

Do Firebase Realtime Database NIE importować zwykłego data.json.

Importować wyłącznie firebase-import.json do ścieżki:
/datavault/live

Po imporcie struktura ma wyglądać tak:

datavault
└── live
    ├── schemaVersion
    ├── createdAt
    ├── source
    └── dataJson

Wartości:

/datavault/live/schemaVersion:
"datavault-firebase-import-v1"

/datavault/live/createdAt:
ISO string, np. "2026-05-12T09:30:00.000Z"

/datavault/live/source:
"Repozytorium.xlsx"

/datavault/live/dataJson:
string zawierający pełny JSON.stringify(dataJsonObject)

Loader aplikacji ma robić:

const snapshotValue = snapshot.val();

if (
  snapshotValue &&
  snapshotValue.schemaVersion === "datavault-firebase-import-v1" &&
  typeof snapshotValue.dataJson === "string"
) {
  return JSON.parse(snapshotValue.dataJson);
}

Dopiero wynik JSON.parse jest właściwym obiektem danych aplikacji.

7. KOD shared/firebase-data-loader.js
======================================================================

Utwórz plik shared/firebase-data-loader.js w takim stylu:

import { initializeApp, getApps, getApp } from "https://www.gstatic.com/firebasejs/12.6.0/firebase-app.js";
import {
  getAuth,
  setPersistence,
  browserLocalPersistence,
  signInWithEmailAndPassword,
  signOut,
  onAuthStateChanged
} from "https://www.gstatic.com/firebasejs/12.6.0/firebase-auth.js";
import {
  getDatabase,
  ref,
  get
} from "https://www.gstatic.com/firebasejs/12.6.0/firebase-database.js";

const FIREBASE_IMPORT_SCHEMA_VERSION = "datavault-firebase-import-v1";
const DATA_PATH = "datavault/live";

const firebaseConfig = {
  apiKey: "UZUPEŁNIJ_Z_FIREBASE_PROJECT_SETTINGS",
  authDomain: "UZUPEŁNIJ_Z_FIREBASE_PROJECT_SETTINGS",
  databaseURL: "https://wh40k-data-slate-default-rtdb.europe-west1.firebasedatabase.app",
  projectId: "UZUPEŁNIJ_Z_FIREBASE_PROJECT_SETTINGS",
  storageBucket: "UZUPEŁNIJ_Z_FIREBASE_PROJECT_SETTINGS",
  messagingSenderId: "UZUPEŁNIJ_Z_FIREBASE_PROJECT_SETTINGS",
  appId: "UZUPEŁNIJ_Z_FIREBASE_PROJECT_SETTINGS"
};

const DATA_ACCESS_EMAIL = "UZUPEŁNIJ_TECHNICZNY_EMAIL_FIREBASE_AUTH";

let app;
let auth;
let database;
let authReadyPromise;

function initFirebaseDataAccess() {
  if (!app) {
    app = getApps().length ? getApp() : initializeApp(firebaseConfig);
    auth = getAuth(app);
    database = getDatabase(app);

    authReadyPromise = setPersistence(auth, browserLocalPersistence)
      .then(() => new Promise((resolve) => {
        const unsubscribe = onAuthStateChanged(auth, (user) => {
          unsubscribe();
          resolve(user);
        });
      }));
  }

  return { app, auth, database };
}

async function waitForAuthReady() {
  initFirebaseDataAccess();
  return authReadyPromise;
}

function getCurrentUser() {
  initFirebaseDataAccess();
  return auth.currentUser;
}

async function loginWithGroupPassword(password) {
  initFirebaseDataAccess();

  const cleanPassword = String(password || "").trim();
  if (!cleanPassword) {
    throw new Error("EMPTY_PASSWORD");
  }

  return signInWithEmailAndPassword(auth, DATA_ACCESS_EMAIL, cleanPassword);
}

async function logoutDataAccess() {
  initFirebaseDataAccess();
  await signOut(auth);
}

function unwrapDataVaultPayload(snapshotValue) {
  if (
    snapshotValue &&
    snapshotValue.schemaVersion === FIREBASE_IMPORT_SCHEMA_VERSION &&
    typeof snapshotValue.dataJson === "string"
  ) {
    try {
      return JSON.parse(snapshotValue.dataJson);
    } catch (error) {
      const wrapped = new Error("FIREBASE_IMPORT_DATAJSON_PARSE_FAILED");
      wrapped.cause = error;
      throw wrapped;
    }
  }

  // Fallback tylko do testów/starej struktury.
  // Produkcyjnie oczekiwany jest wrapper datavault-firebase-import-v1.
  return snapshotValue;
}

async function loadDataVaultLive() {
  initFirebaseDataAccess();

  const user = await waitForAuthReady();
  if (!user) {
    throw new Error("NOT_AUTHENTICATED");
  }

  const dataRef = ref(database, DATA_PATH);
  const snapshot = await get(dataRef);

  if (!snapshot.exists()) {
    throw new Error("DATA_NOT_FOUND");
  }

  return unwrapDataVaultPayload(snapshot.val());
}

function getReadableAccessError(error, lang = "pl") {
  const code = String((error && (error.code || error.message)) || "");

  if (code.includes("EMPTY_PASSWORD")) {
    return lang === "en" ? "Enter the access password." : "Podaj hasło dostępu.";
  }

  if (
    code.includes("auth/invalid-credential") ||
    code.includes("auth/wrong-password") ||
    code.includes("auth/user-not-found")
  ) {
    return lang === "en" ? "Invalid access password." : "Nieprawidłowe hasło dostępu.";
  }

  if (code.includes("auth/too-many-requests")) {
    return lang === "en"
      ? "Too many failed attempts. Try again later."
      : "Zbyt wiele błędnych prób. Spróbuj później.";
  }

  if (
    code.includes("permission_denied") ||
    code.includes("PERMISSION_DENIED") ||
    code.includes("permission-denied")
  ) {
    return lang === "en"
      ? "No permission to read private data."
      : "Brak uprawnień do odczytu prywatnych danych.";
  }

  if (code.includes("NOT_AUTHENTICATED")) {
    return lang === "en"
      ? "Sign in to access private data."
      : "Zaloguj się, aby uzyskać dostęp do prywatnych danych.";
  }

  if (code.includes("DATA_NOT_FOUND")) {
    return lang === "en"
      ? "Private data was not found in Firebase."
      : "Nie znaleziono prywatnych danych w Firebase.";
  }

  if (code.includes("FIREBASE_IMPORT_DATAJSON_PARSE_FAILED")) {
    return lang === "en"
      ? "Firebase data wrapper is invalid. Cannot parse dataJson."
      : "Wrapper danych w Firebase jest niepoprawny. Nie można sparsować dataJson.";
  }

  return lang === "en"
    ? "Could not load private data."
    : "Nie udało się załadować prywatnych danych.";
}

window.DataVaultFirebase = {
  initFirebaseDataAccess,
  waitForAuthReady,
  getCurrentUser,
  loginWithGroupPassword,
  logoutDataAccess,
  loadDataVaultLive,
  unwrapDataVaultPayload,
  getReadableAccessError
};

window.DataVaultFirebaseReady = Promise.resolve(window.DataVaultFirebase);

12. ZMIANY W DataVault/app.js — USUNIĘCIE LOKALNEGO data.json
======================================================================

Znajdź funkcję, która obecnie ładuje dane z data.json. Może nazywać się:
- loadJson
- loadData
- init
- start
- fetchData
albo podobnie.

Usuń lub wyłącz:

fetch("data.json")
fetch("./data.json")
fetch("../DataVault/data.json")
fetch(DATA_URL) wskazujący lokalny data.json

Zastąp tym:

async function loadPrivateDataFromFirebase() {
  const firebaseApi = await window.DataVaultFirebaseReady;
  setStatus(translations[currentLanguage].messages.statusLoadJson);

  const data = await firebaseApi.loadDataVaultLive();

  DB = normaliseDB(data);
  const restored = loadSessionState();
  if (!restored) {
    applyDefaultViewToAllSheetsIfCurrentCodeHasSuchFunction();
  }

  initUI();
  setStatus(translations[currentLanguage].messages.statusLoadOk);
}

Dopasuj nazwy funkcji do istniejącej architektury.
Najważniejsze:
- DB ma dostać normaliseDB(data),
- initUI/render tabs/render body ma odpalić się po pobraniu danych,
- nie próbować renderować tabel przed pobraniem danych,
- nie ładować data.json z hostingu.

Jeżeli istnieje dotychczasowa funkcja:

fetch("data.json")
  .then(r => r.json())
  .then(data => { DB = normaliseDB(data); initUI(); })

ma zostać zastąpiona przez:

const firebaseApi = await window.DataVaultFirebaseReady;
const data = await firebaseApi.loadDataVaultLive();
DB = normaliseDB(data);
initUI();

function showAccessGate(message = "") {
  if (els.accessGate) els.accessGate.hidden = false;
  if (els.accessError) els.accessError.textContent = message;
}

function hideAccessGate() {
  if (els.accessGate) els.accessGate.hidden = true;
  if (els.accessError) els.accessError.textContent = "";
  if (els.accessPassword) els.accessPassword.value = "";
}

function clearRuntimeData() {
  DB = null;
  currentSheet = null;
  // wyczyść tabs/table/select/porównanie w sposób zgodny z obecnym kodem
  if (els.tabs) els.tabs.innerHTML = "";
  if (els.wrap) {
    els.wrap.innerHTML = `
      <div class="emptyState">
        <div class="emptyTitle">${translations[currentLanguage].labels.emptyTitle}</div>
        <div class="emptyText">${translations[currentLanguage].labels.emptyText}</div>
      </div>
    `;
  }
}

async function startPrivateDataFlow() {
  const firebaseApi = await window.DataVaultFirebaseReady;

  try {
    await firebaseApi.initFirebaseDataAccess();
    const user = await firebaseApi.waitForAuthReady();

    if (!user) {
      clearRuntimeData();
      showAccessGate();
      return;
    }

    hideAccessGate();
    await loadPrivateDataFromFirebase();
  } catch (error) {
    clearRuntimeData();
    showAccessGate(firebaseApi.getReadableAccessError(error, currentLanguage));
  }
}

15. ZMIANY W GeneratorNPC — NIE UŻYWAĆ GOŁEGO REST FETCH
======================================================================

Obecne częściowe wdrożenie może mieć:

const DATA_URL = "https://wh40k-data-slate-default-rtdb.europe-west1.firebasedatabase.app/datavault/live.json";

oraz potem:
fetch(DATA_URL)

To jest niepoprawne jako docelowe zabezpieczenie.

Usuń:
- const DATA_URL = ".../datavault/live.json"
- fetch(DATA_URL)
- response.json() jako bezpośrednie dane GeneratorNPC

Zastąp wspólnym loaderem:

const firebaseApi = await window.DataVaultFirebaseReady;
const data = await firebaseApi.loadDataVaultLive();

Następnie przekazuj data do dotychczasowej logiki GeneratorNPC.

Jeżeli GeneratorNPC ma funkcję:
async function loadData() { ... }

zmień ją tak:

async function loadData() {
  const firebaseApi = await window.DataVaultFirebaseReady;
  setDataStatus("Ładowanie danych z prywatnej bazy...");

  try {
    const data = await firebaseApi.loadDataVaultLive();
    // dalej istniejące przetwarzanie:
    // state.data = data;
    // buildIndexes(data);
    // populateSelects();
    // render...
    setDataStatus("OK — załadowano dane z prywatnej bazy");
  } catch (error) {
    const user = firebaseApi.getCurrentUser();
    if (!user) {
      showAccessGate();
      return;
    }
    setDataStatus(firebaseApi.getReadableAccessError(error, currentLanguage));
  }
}

16. BRAMKA HASŁA W GeneratorNPC
======================================================================

GeneratorNPC również musi mieć ekran hasła albo korzystać ze wspólnego komponentu.

Minimalne wymaganie:
- jeżeli użytkownik nie jest zalogowany, GeneratorNPC pokazuje ekran hasła,
- po poprawnym haśle ładuje dane przez loadDataVaultLive(),
- jeżeli użytkownik zalogował się wcześniej w DataVault na tym samym originie, GeneratorNPC nie pyta ponownie.

17. ROZPAKOWANIE WRAPPERA W OBU MODUŁACH
======================================================================

Rozpakowanie ma być tylko w shared/firebase-data-loader.js.

Nie dublować logiki w DataVault i GeneratorNPC.

Centralna funkcja:
unwrapDataVaultPayload(snapshotValue)

ma obsługiwać:

{
  "schemaVersion": "datavault-firebase-import-v1",
  "createdAt": "...",
  "source": "Repozytorium.xlsx",
  "dataJson": "{...}"
}

i zwracać:
JSON.parse(snapshotValue.dataJson)

DataVault i GeneratorNPC mają dostać już rozpakowany obiekt.
