0. CEL TEGO ZADANIA
======================================================================

Napraw błąd logowania w module GeneratorNPC:

Firebase Authentication nie jest poprawnie skonfigurowane.

Obecna sytuacja:
- DataVault działa poprawnie.
- Logowanie w DataVault działa.
- Dane z Firebase Realtime Database ładują się poprawnie w DataVault.
- GeneratorNPC po wejściu pokazuje bramkę logowania.
- GeneratorNPC nie korzysta z sesji uzyskanej wcześniej w DataVault.
- Po wpisaniu hasła w GeneratorNPC pojawia się błąd:
  „Firebase Authentication nie jest poprawnie skonfigurowane.”

Diagnoza:
GeneratorNPC miesza dwa różne projekty Firebase:
1. Projekt prywatnych danych DataVault:
   wh40k-data-slate

2. Stary projekt GeneratorNPC / favorites:
   audiorpg-2eb6f

shared/firebase-data-loader.js używa aktualnie niebezpiecznej logiki:

app = getApps().length ? getApp() : initializeApp(firebaseConfig);

To jest błąd.

Jeżeli GeneratorNPC wcześniej zainicjalizuje domyślną aplikację Firebase dla projektu audiorpg-2eb6f, to shared/firebase-data-loader.js bierze tę istniejącą domyślną aplikację przez getApp() i próbuje logować użytkownika taryf@o2.pl w złym projekcie Firebase.

Efekt:
- DataVault działa, bo trafia w dobry projekt.
- GeneratorNPC nie działa, bo prywatny loader może używać złego Firebase App.
- Komunikat „auth/configuration-not-found” / „Firebase Authentication nie jest poprawnie skonfigurowane” wynika z użycia złego projektu Firebase.

Docelowo:
- prywatne dane DataVault muszą mieć własną nazwaną aplikację Firebase,
- GeneratorNPC favorites, jeśli nadal potrzebne, musi mieć osobną nazwaną aplikację Firebase,
- żaden z tych mechanizmów nie może przejmować przypadkowej aplikacji domyślnej przez getApp(),
- DataVault i GeneratorNPC muszą używać tej samej nazwanej aplikacji dla prywatnych danych,
- sesja Auth musi być współdzielona między DataVault i GeneratorNPC.

1. NAJWAŻNIEJSZA ZMIANA: NIE UŻYWAĆ getApp() DLA PRYWATNYCH DANYCH
======================================================================

W pliku:

shared/firebase-data-loader.js

znajdź kod podobny do:

app = getApps().length ? getApp() : initializeApp(firebaseConfig);

Usuń tę logikę.

Zastąp ją nazwanym Firebase App dla prywatnych danych.

Dodaj stałą:

const PRIVATE_DATA_APP_NAME = "wh40k-data-slate-private-data";

Dodaj funkcję:

function getPrivateDataApp() {
  const firebaseConfig = getFirebaseConfig();

  const existing = getApps().find((candidate) => candidate.name === PRIVATE_DATA_APP_NAME);

  if (existing) {
    return existing;
  }

  return initializeApp(firebaseConfig, PRIVATE_DATA_APP_NAME);
}

Następnie w initFirebaseDataAccess() użyj:

app = getPrivateDataApp();
auth = getAuth(app);
database = getDatabase(app);

Czyli cała inicjalizacja ma wyglądać koncepcyjnie tak:

function initFirebaseDataAccess() {
  if (!app) {
    assertFirebaseRuntimeConfig();

    app = getPrivateDataApp();
    auth = getAuth(app);
    database = getDatabase(app);

    authReadyPromise = setPersistence(auth, browserLocalPersistence)
      .catch((error) => {
        console.warn("Firebase Auth persistence warning:", error);
      })
      .then(() => new Promise((resolve) => {
        let resolved = false;

        authUnsubscribe = onAuthStateChanged(auth, (user) => {
          currentAuthUser = user || null;

          if (!resolved) {
            resolved = true;
            resolve(currentAuthUser);
          }
        });
      }));
  }

  return { app, auth, database };
}

Ważne:
- Nie używaj getApp() bez nazwy.
- Nie używaj getApps().length jako testu dla prywatnych danych.
- Prywatny loader zawsze ma używać aplikacji nazwanej:
  wh40k-data-slate-private-data
- Ten sam app name musi być użyty zarówno w DataVault, jak i GeneratorNPC.

function getOrCreateNamedFirebaseApp(name, config) {
  const existing = getApps().find((candidate) => candidate.name === name);

  if (existing) {
    return existing;
  }

  return initializeApp(config, name);

12. NIE BLOKOWAĆ DANYCH NPC PRZEZ AWARIĘ FAVORITES
======================================================================

Jeżeli stary projekt favorites ma problem, GeneratorNPC nie może wisieć na „Ładowanie danych...” ani blokować prywatnych danych.

Zasada:
- awaria DataVault private loader = blokuje dane NPC, pokazuje bramkę/błąd,
- awaria favorites = pokazuje błąd tylko w panelu Ulubione, ale nie blokuje ładowania danych NPC.

Jeżeli obecny kod inicjalizuje favorites przed ładowaniem danych NPC i wyjątek favorites przerywa cały moduł, przenieś inicjalizację favorites do osobnego try/catch.

Przykład:

try {
  initFavorites();
} catch (error) {
  console.error("[GeneratorNPC] Favorites init failed:", error);
  setFavoritesStatus("Nie udało się połączyć z ulubionymi.");
}

await startPrivateDataFlow();

Nie pozwól, aby błąd favorites zatrzymał startPrivateDataFlow().
