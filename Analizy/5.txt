======================================================================
0. CEL TEGO ZADANIA
======================================================================

Napraw błąd logowania do prywatnych danych Firebase w DataVault.

Obecny objaw w aplikacji:

Po otwarciu DataVault pojawia się bramka:

Dostęp do prywatnych danych
Dane aplikacji są prywatne. Podaj hasło dostępu.

Po wpisaniu poprawnego hasła aplikacja pokazuje:

Zaloguj się, aby uzyskać dostęp do prywatnych danych.

W konsoli Edge widać:

Tryb: ADMIN
Ładowanie danych z prywatnej bazy...
Zaloguj się, aby uzyskać dostęp do prywatnych danych.
BŁĄD LOGOWANIA FIREBASE: Zaloguj się, aby uzyskać dostęp do prywatnych danych.

To oznacza, że aplikacja rzuca błąd:

NOT_AUTHENTICATED

po próbie logowania.

Najważniejsza diagnoza:
To nie wygląda już na problem z hasłem, mailem, UID ani strukturą Firebase Database.
Problem jest w logice shared/firebase-data-loader.js.

Obecny loader zapamiętuje pierwszy stan Auth jako null i potem nadal używa tego starego wyniku, nawet po udanym signInWithEmailAndPassword().

======================================================================
1. OBECNY BŁĘDNY MECHANIZM
======================================================================

W shared/firebase-data-loader.js obecnie istnieje logika podobna do:

let app, auth, database, authReadyPromise;

function initFirebaseDataAccess(){
  if(!app){
    ...
    authReadyPromise = setPersistence(auth, browserLocalPersistence).then(()=>new Promise((resolve)=>{
      const unsubscribe = onAuthStateChanged(auth, (user)=>{
        unsubscribe();
        resolve(user);
      });
    }));
  }
  return {app,auth,database};
}

async function waitForAuthReady(){
  initFirebaseDataAccess();
  return authReadyPromise;
}

async function loginWithGroupPassword(password){
  initFirebaseDataAccess();
  ...
  return signInWithEmailAndPassword(auth,email,clean);
}

async function loadDataVaultLive(){
  initFirebaseDataAccess();
  const user=await waitForAuthReady();
  if(!user) throw new Error('NOT_AUTHENTICATED');
  ...
}

Problem:

1. Przy pierwszym starcie użytkownik nie jest zalogowany.
2. onAuthStateChanged zwraca user = null.
3. authReadyPromise rozwiązuje się jako null.
4. Loader natychmiast unsubscribe() z obserwatora.
5. Użytkownik wpisuje poprawne hasło.
6. signInWithEmailAndPassword może się udać.
7. loadDataVaultLive() nadal robi await waitForAuthReady().
8. waitForAuthReady() nadal zwraca stare authReadyPromise, czyli null.
9. loadDataVaultLive() rzuca NOT_AUTHENTICATED.
10. UI pokazuje „Zaloguj się, aby uzyskać dostęp do prywatnych danych.”

To jest błąd stanu Auth w loaderze.

======================================================================
2. GŁÓWNA POPRAWKA
======================================================================

Popraw shared/firebase-data-loader.js tak, żeby:

1. onAuthStateChanged NIE był odpinany po pierwszym wywołaniu.
2. Loader trzymał aktualnego użytkownika w zmiennej currentAuthUser.
3. Pierwszy stan Auth służył tylko do informacji, że Firebase Auth jest gotowe.
4. Po signInWithEmailAndPassword() currentAuthUser był ustawiany na credential.user.
5. waitForAuthReady() zwracał aktualnego użytkownika, a nie stare null.
6. loadDataVaultLive() sprawdzał auth.currentUser lub currentAuthUser po logowaniu.
7. Błąd NOT_AUTHENTICATED występował tylko wtedy, gdy naprawdę nie ma aktywnego użytkownika.

======================================================================
3. ZMODYFIKUJ shared/firebase-data-loader.js — ZMIENNE STANU
======================================================================

W shared/firebase-data-loader.js zastąp:

let app, auth, database, authReadyPromise;

wersją:

let app;
let auth;
let database;
let authReadyPromise;
let authUnsubscribe = null;
let currentAuthUser = null;

Znaczenie:
- currentAuthUser przechowuje aktualny stan użytkownika Firebase Auth.
- authReadyPromise mówi tylko, że pierwszy odczyt Auth został wykonany.
- authUnsubscribe pozwala w przyszłości odpiąć listener, jeśli będzie taka potrzeba.
- Nie odpinamy listenera od razu po pierwszym wywołaniu.

======================================================================
4. ZMODYFIKUJ initFirebaseDataAccess()
======================================================================

Obecnie listener robi unsubscribe() od razu po pierwszym stanie.
To jest źródło problemu.

Zmień initFirebaseDataAccess() na strukturę podobną do tej:

function initFirebaseDataAccess(){
  if(!app){
    assertFirebaseRuntimeConfig();

    const firebaseConfig = getFirebaseConfig();

    app = getApps().length ? getApp() : initializeApp(firebaseConfig);
    auth = getAuth(app);
    database = getDatabase(app);

    authReadyPromise = setPersistence(auth, browserLocalPersistence)
      .catch((error) => {
        console.warn("Firebase Auth persistence warning:", error);
      })
      .then(() => new Promise((resolve) => {
        let resolved = false;

        authUnsubscribe = onAuthStateChanged(auth, (user) => {
          currentAuthUser = user || null;

          if (!resolved) {
            resolved = true;
            resolve(currentAuthUser);
          }
        });
      }));
  }

  return { app, auth, database };
}

Ważne:
- Usuń unsubscribe() z wnętrza onAuthStateChanged.
- Listener ma działać cały czas i aktualizować currentAuthUser.
- Promise ma rozwiązać się tylko raz przy pierwszym stanie.
- Kolejne zmiany Auth mają aktualizować currentAuthUser, ale nie muszą ponownie rozwiązywać Promise.
- setPersistence nie powinien całkowicie zabić aplikacji bez czytelnego błędu. Można zalogować warning, ale zachować mechanizm logowania.

Uwaga o Edge:
W konsoli może pojawiać się ostrzeżenie:
Tracking Prevention blocked access to storage for <URL>.

Nie traktować tego automatycznie jako błędu hasła. To może wpływać na trwałość sesji, ale nie powinno uniemożliwiać samego logowania w bieżącej karcie. Jeżeli browserLocalPersistence sprawia problemy, można później dodać fallback do inMemoryPersistence, ale główny błąd teraz to stale authReadyPromise = null.

======================================================================
5. ZMODYFIKUJ waitForAuthReady()
======================================================================

Zmień funkcję na:

async function waitForAuthReady(){
  initFirebaseDataAccess();

  await authReadyPromise;

  currentAuthUser = auth.currentUser || currentAuthUser || null;

  return currentAuthUser;
}

Nie zwracaj bezpośrednio starego authReadyPromise jako wyniku użytkownika.
Najpierw poczekaj, aż Auth będzie gotowe, potem pobierz aktualne auth.currentUser.

======================================================================
6. ZMODYFIKUJ getCurrentUser()
======================================================================

Zmień funkcję na:

function getCurrentUser(){
  initFirebaseDataAccess();
  return auth.currentUser || currentAuthUser || null;
}

Dzięki temu inne części aplikacji dostają aktualny stan po loginie.

======================================================================
7. ZMODYFIKUJ loginWithGroupPassword()
======================================================================

Obecnie loginWithGroupPassword() zwraca signInWithEmailAndPassword(), ale nie aktualizuje currentAuthUser ani authReadyPromise.

Zmień na:

async function loginWithGroupPassword(password){
  initFirebaseDataAccess();

  const clean = String(password || "").trim();

  if(!clean) {
    throw new Error("EMPTY_PASSWORD");
  }

  const email = getDataAccessEmail();

  if(!email) {
    throw new Error("MISSING_DATA_ACCESS_EMAIL");
  }

  const credential = await signInWithEmailAndPassword(auth, email, clean);

  currentAuthUser = credential.user || auth.currentUser || null;
  authReadyPromise = Promise.resolve(currentAuthUser);

  if (currentAuthUser && typeof currentAuthUser.getIdToken === "function") {
    await currentAuthUser.getIdToken(true);
  }

  return credential;
}

Ważne:
- Po udanym signInWithEmailAndPassword() od razu zapisz currentAuthUser.
- Nadpisz authReadyPromise na Promise.resolve(currentAuthUser), żeby kolejne loadDataVaultLive() nie widziało starego null.
- getIdToken(true) jest dodatkową defensywną synchronizacją tokenu przed odczytem RTDB.
- Nie zapisuj hasła nigdzie.

======================================================================
8. ZMODYFIKUJ logoutDataAccess()
======================================================================

Po wylogowaniu trzeba wyczyścić currentAuthUser i authReadyPromise.

Zmień na:

async function logoutDataAccess(){
  initFirebaseDataAccess();

  await signOut(auth);

  currentAuthUser = null;
  authReadyPromise = Promise.resolve(null);
}

======================================================================
9. ZMODYFIKUJ loadDataVaultLive()
======================================================================

Obecnie loadDataVaultLive() robi:

const user = await waitForAuthReady();
if(!user) throw new Error('NOT_AUTHENTICATED');

To może dalej zwrócić stare null, jeśli waitForAuthReady() nie został poprawiony.

Po poprawkach użyj wersji defensywnej:

async function loadDataVaultLive(){
  initFirebaseDataAccess();

  await waitForAuthReady();

  const user = auth.currentUser || currentAuthUser || null;

  if(!user) {
    throw new Error("NOT_AUTHENTICATED");
  }

  if (typeof user.getIdToken === "function") {
    await user.getIdToken();
  }

  const snapshot = await get(ref(database, DATA_PATH));

  if(!snapshot.exists()) {
    throw new Error("DATA_NOT_FOUND");
  }

  return unwrapDataVaultPayload(snapshot.val());
}

Ważne:
- Nie polegaj wyłącznie na wyniku authReadyPromise.
- Sprawdź auth.currentUser i currentAuthUser po udanym loginie.
- Dopiero potem pobieraj Realtime Database.

======================================================================

Na czas diagnozy dodaj bezpieczne logi bez hasła.

Możesz dodać funkcję:

function debugAuthState(label){
  const user = auth ? (auth.currentUser || currentAuthUser) : null;
  console.info("[DataVaultFirebase]", label, {
    hasAuth: !!auth,
    hasCurrentUser: !!user,
    uid: user ? user.uid : null,
    email: user ? user.email : null
  });
}

Użyj jej w:
- initFirebaseDataAccess() po onAuthStateChanged,
- loginWithGroupPassword() po signInWithEmailAndPassword,
- loadDataVaultLive() przed sprawdzeniem NOT_AUTHENTICATED.

Przykład:

debugAuthState("after login");
debugAuthState("before loadDataVaultLive");

Nie loguj hasła.
Nie loguj dataJson.
Nie loguj pełnych danych z bazy.

======================================================================

Obecny komunikat przy NOT_AUTHENTICATED jest poprawny, ale przy debugowaniu warto rozróżnić:
- błąd logowania Auth,
- brak sesji po loginie,
- permission denied z RTDB.

Upewnij się, że getReadableAccessError() ma:

if(code.includes("auth/invalid-credential") || code.includes("auth/wrong-password") || code.includes("auth/user-not-found")) {
  return lang === "en" ? "Invalid access password." : "Nieprawidłowe hasło dostępu.";

if(code.includes("auth/invalid-api-key")) {
  return lang === "en" ? "Invalid Firebase apiKey." : "Nieprawidłowy apiKey Firebase.";

if(code.includes("auth/configuration-not-found")) {
  return lang === "en" ? "Firebase Authentication is not configured correctly." : "Firebase Authentication nie jest poprawnie skonfigurowane.";

if(code.includes("NOT_AUTHENTICATED")) {

if(code.includes("permission_denied") || code.includes("PERMISSION_DENIED") || code.includes("permission-denied")) {

12. SPRAWDŹ DataVault/app.js PO POPRAWCE LOADERA
======================================================================

W DataVault/app.js sprawdź, czy obsługa formularza robi:

await firebaseApi.loginWithGroupPassword(els.accessPassword?.value || "");
hideAccessGate();
await loadPrivateDataFromFirebase();

To jest OK, ale po poprawce loadera loadPrivateDataFromFirebase() nie może już dostawać NOT_AUTHENTICATED po udanym loginie.

Sprawdź też, czy app.js używa getFirebaseApi(), a nie bezpośrednio window.DataVaultFirebaseReady, np.:

const firebaseApi = await getFirebaseApi();

To zostało wcześniej zalecone i powinno pozostać.

Nie zmieniaj tutaj logiki generatora data.json/firebase-import.json.

20. PODSUMOWANIE DLA AGENTA
======================================================================

Obecny błąd nie polega na tym, że Firebase Database jest źle zaimportowane.
Obecny błąd nie polega już na placeholderach w configu.
Obecny błąd nie wygląda na błędne hasło.

Problemem jest stan Firebase Auth w shared/firebase-data-loader.js.

Loader zapamiętał początkowy brak użytkownika jako null i po poprawnym loginie nadal używa tego starego null.
