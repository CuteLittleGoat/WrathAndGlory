2) Zakres i metoda

  • Przeanalizowano konfigurację PWA, Service Workera, punkty subskrypcji/triggera push oraz odwołania do Cloudflare Worker.
  • Przeanalizowano warunki wdrożenia aplikacji Android (WebView vs TWA) pod kątem wymagań 1–10.

3) Najważniejsze ustalenia techniczne (stan obecny repo)

  • Main posiada przycisk aktywacji powiadomień, rejestruje Service Workera i zapisuje subskrypcję push na endpoint subscribeEndpoint.
  • GM (Infoczytnik/GM.html) po kliknięciu „Wyślij” odpala endpoint triggerEndpoint, który ma wysłać push do subskrybentów.
  • Service Worker obsługuje zdarzenia push i notificationclick, z domyślnym body: +++ INCOMING DATA-TRANSMISSION +++ oraz ikoną /IkonaPowiadomien.png.
  • Manifest PWA wskazuje ikonę główną IkonaGlowna.png i start URL ./Main/index.html?pwa=1.
  • Konfiguracja push wskazuje na Cloudflare Workers: wrathandglory-push-api.tarczynski-pawel.workers.dev (subscribe/trigger).

4) Ocena wymagań 1–10 (Android Studio)

  1. Tylko widok użytkownika (bez ?admin=1): możliwe i proste. Aplikacja mobilna powinna wymuszać URL bez parametru admin i blokować nawigację do adresów zawierających admin=1.
  2. Infoczytnik zawsze pionowo: technicznie możliwe natywnie po stronie Androida (ustawienie orientacji per ekran/URL).
  3. Pozostałe moduły pion/poziom: możliwe natywnie po stronie Androida przez dynamiczną zmianę orientacji poza Infoczytnikiem.
  4. Logo aplikacji = IkonaGlowna.png: możliwe (adaptive icon / mipmap) bez problemu.
  5. Wsparcie powiadomień Android: możliwe, najlepiej przez natywne FCM (największa kontrola i przewidywalność).
  6. Tylko online: możliwe — wyłączyć cache offline w aplikacji Android i wymuszać połączenie sieciowe.
  7. Ikona powiadomień = IkonaPowiadomien.png: możliwe (po konwersji do zasobów notyfikacji Android, najlepiej monochromatyczna small icon + opcjonalna large icon).
  8. Stały tekst powiadomienia: możliwe, tekst ustawiany po stronie natywnej lub serwera push.
  9. Trigger z panelu GM: obecna architektura już to wspiera przez endpoint triggera (Cloudflare Worker). Można utrzymać ten mechanizm.
  10. A/B/C (foreground/background/killed): najpewniejsze spełnienie daje natywne FCM. Czysty web push (PWA/TWA) bywa bardziej zależny od polityk systemu/przeglądarki.

5) Czy da się użyć obecnej konfiguracji Cloudflare w projekcie Android Studio?

Tak, częściowo / warunkowo:

  • Jeśli zostaje model web push (PWA/TWA), obecne endpointy Cloudflare /api/push/subscribe i /api/push/trigger można wykorzystać dalej.
  • Jeśli celem jest gwarancja powiadomień A/B/C, rekomendowane jest rozszerzenie Workerów o kanał natywny FCM (przechowywanie tokenów urządzeń Android + wysyłka FCM).
  • W praktyce: jeden backend (Cloudflare Worker) może obsłużyć oba typy odbiorców: subskrypcje web push i tokeny natywne FCM.

6) WebView vs TWA — porównanie

WebView

  • Zalety: pełna kontrola natywna (orientacja per moduł/URL, blokada admin, lifecycle, integracja z FCM i kanałami notyfikacji).
  • Wady: większy koszt implementacji i utrzymania; część funkcji PWA nie działa jak w Chrome; trzeba dopilnować bezpieczeństwa WebView.

TWA (Trusted Web Activity)

  • Zalety: szybkie opakowanie istniejącej PWA, renderowanie przez Chrome, naturalne wsparcie standardów web/PWA.
  • Wady: mniejsza kontrola natywna (zwłaszcza orientacja per pojedynczy moduł/ścieżka), zależność od Chrome i warunków Digital Asset Links.

7) Rekomendacja docelowa (gwarancja funkcjonalności)

Rekomendacja: Android app oparta o WebView + natywne FCM + istniejący Cloudflare Worker rozszerzony o obsługę tokenów FCM.

  • To rozwiązanie najłatwiej spełnia jednocześnie wymogi: 1 (brak admin), 2/3 (orientacja per moduł), 5/7/8/9/10 (powiadomienia z pełną kontrolą).
  • PWA i Service Worker mogą nadal działać dla wersji webowej, ale na Androidzie krytyczne notyfikacje opierać o natywne FCM.

8) Proponowana architektura wdrożenia

  1. Android Studio: jedna aplikacja z WebView, start od /Main/index.html bez parametru admin.
  2. Przechwytywanie URL w WebView: jeżeli URL zawiera admin=1, przekierowanie do wersji user-only.
  3. Dynamiczna orientacja: dla URL zawierających /Infoczytnik/Infoczytnik.html — portrait; dla pozostałych — unspecified/user.
  4. Ikona aplikacji: IkonaGlowna.png.
  5. Firebase/FCM w aplikacji: zapis tokenu urządzenia do Cloudflare Worker (nowy endpoint np. /api/android/register).
  6. GM „Wyślij” uruchamia trigger (już jest): Worker wysyła zarówno web push, jak i natywne FCM.
  7. Payload FCM: stały body +++ INCOMING DATA-TRANSMISSION +++, ikona notyfikacji z zasobu opartego o IkonaPowiadomien.png.
  8. Tryb online-only: brak cache offline w Androidzie + komunikat braku sieci.

9) Ryzyka i uwagi

  • Android small notification icon ma specyficzne wymagania (najlepiej biała, przezroczysta); może być potrzebna wersja uproszczona ikony.
  • „Gwarancja 100%” dostarczeń push nigdy nie jest absolutna (OS/battery/network), ale natywne FCM minimalizuje ryzyko względem czystego web push.
  • Jeżeli pozostanie TWA + web push bez natywnego FCM, wariant C (app killed) jest bardziej podatny na ograniczenia środowiskowe.

10.1 Cloudflare — co trzeba zrobić krok po kroku

  1. Potwierdź obecny Worker i bindingi: utrzymaj endpointy już działające (/api/push/health, /api/push/subscribe, /api/push/trigger) oraz KV SUBSCRIPTIONS_KV do web push.
  2. Dodaj nowe KV na tokeny Android/FCM: utwórz np. FCM_TOKENS_KV i podepnij binding do tego samego Workera.
  3. Rozszerz kod Workera o endpoint rejestracji tokenu FCM: np. POST /api/android/register z body zawierającym fcmToken, deviceId, platform, appVersion. Token zapisz w FCM_TOKENS_KV pod stabilnym kluczem (np. hash tokenu albo deviceId).
  4. Dodaj endpoint wyrejestrowania: np. POST /api/android/unregister usuwający token przy wylogowaniu/odinstalowaniu.
  5. Rozszerz istniejący trigger: w /api/push/trigger po web push dodaj wysyłkę do wszystkich tokenów FCM (pętla po FCM_TOKENS_KV), z usuwaniem tokenów nieważnych (np. UNREGISTERED).
  6. Dodaj sekcję sekretów Firebase do Workera:
    • FIREBASE_PROJECT_ID
    • FIREBASE_CLIENT_EMAIL
    • FIREBASE_PRIVATE_KEY (zachowaj format wieloliniowy, zwykle z \n w sekretach)
  7. Dodaj logikę OAuth2 dla FCM HTTP v1: Worker generuje JWT service account, pobiera access token z Google OAuth (https://oauth2.googleapis.com/token), a następnie wywołuje https://fcm.googleapis.com/v1/projects/{PROJECT_ID}/messages:send.
  8. Zdefiniuj payload FCM zgodny z wymaganiem:
    • notification.title np. „Infoczytnik”
    • notification.body = +++ INCOMING DATA-TRANSMISSION +++
    • android.notification.channel_id np. infoczytnik_messages
    • android.notification.icon = nazwa zasobu small icon po stronie aplikacji Android
    • data.url = docelowo /Infoczytnik/Infoczytnik.html (otwarcie właściwej zakładki po tapnięciu)
  9. Zabezpiecz endpointy Android: analogicznie jak TRIGGER_TOKEN dodaj np. ANDROID_REGISTER_TOKEN, żeby rejestracja/usuwanie tokenów nie była publiczna.
  10. Wdróż i przetestuj na Cloudflare: po deploy sprawdź kolejno health, register, trigger oraz logi Workera (czy idzie i web push, i FCM).

10.2 Firebase — co trzeba zrobić krok po kroku

  1. Włącz Cloud Messaging: w projekcie Firebase aktywuj FCM (zakładka Cloud Messaging).
  2. Dodaj aplikację Android do projektu: podaj poprawny applicationId i pobierz google-services.json do projektu Android Studio.
  3. Wygeneruj konto serwisowe: utwórz/wybierz service account z uprawnieniami do wysyłki FCM (rola typu Firebase Admin / Cloud Messaging odpowiednia dla HTTP v1).
  4. Pobierz klucz JSON service account: z pliku pobierz pola project_id, client_email, private_key i wpisz je do sekretów Cloudflare Workera.
  5. Sprawdź zgodność Project ID: project_id z klucza musi być tym samym, którego Worker użyje w URL FCM v1.
  6. Skonfiguruj aplikację Android pod FCM: zależności Firebase Messaging, utworzenie Notification Channel, pobranie tokenu przez FirebaseMessaging.getInstance().token i wysłanie tokenu do /api/android/register.
  7. Obsłuż odświeżanie tokenu: w FirebaseMessagingService.onNewToken() ponowna rejestracja tokenu w Workerze.
  8. Obsłuż przychodzące wiadomości: przy foreground wyświetl lokalną notyfikację; przy background/killed użyj payloadu FCM tak, by Android wyświetlił systemową notyfikację.
  9. Przetestuj 3 scenariusze A/B/C: aktywna app, app w tle, app ubita — każdy test wywołuj przez realny trigger z panelu GM.

Wniosek dla pytania „czy wszystko jest gotowe do szykowania plików do Android Studio?”
Tak — backend i konfiguracja chmurowa są gotowe, więc można zaczynać przygotowanie plików Android Studio.
Nie — nie wszystko jest gotowe do budowy APK/AAB, bo brakuje jeszcze samej implementacji klienta Android i pierwszej skutecznej rejestracji tokenu FCM.

Minimalny zestaw plików/konfiguracji, który trzeba teraz przygotować w Android Studio:

  1. app/google-services.json (już pobrany — trzeba włożyć do modułu app).
  2. Gradle (project + app): plugin Google Services + zależność Firebase Messaging.
  3. AndroidManifest.xml: internet, receiver/service FCM, intent-filter dla otwierania ekranu Infoczytnika.
  4. Klasa FirebaseMessagingService:
    • onNewToken() → POST do /api/fcm/register,
    • onMessageReceived() → notyfikacja lokalna dla foreground + obsługa silent.
  5. Notification Channel (np. default lub infoczytnik_messages) + zasób res/drawable/ic_notification.xml.
  6. Obsługa kliknięcia notyfikacji (PendingIntent / deep link) tak, by otwierać właściwy ekran WebView.
  7. Konfiguracja WebView (online-only), oraz przekazywanie URL z payloadu (/Infoczytnik/Infoczytnik.html).

Checkpoint „GO do budowania”: gdy /api/push/health pokaże fcmTokens >= 1, a test A/B/C (foreground/background/killed) przejdzie na urządzeniu fizycznym.

10.4 Odpowiedź na pytanie: checkbox w GM.html (z powiadomieniem / bez powiadomienia)

Pytanie użytkownika: "Czy w pliku GM.html da się zrobić jakiś checkbox, którego zaznaczenie lub odznaczenie skutkowałoby wysłaniem powiadomienia lub samej wiadomości bez powiadomienia?"

Tak, da się. W Infoczytnik/GM.html można dodać checkbox (np. Wyślij bez powiadomienia), a jego stan przesłać w payloadzie triggera jako flaga, np. silent: true/false.

  • Wariant 1 (rekomendowany): checkbox ustawia flagę silent; Cloudflare Worker przekazuje ją dalej; Service Worker po stronie odbiorcy przy silent=true aktualizuje dane/wiadomość, ale nie wywołuje showNotification().
  • Wariant 2: Worker wysyła dwa typy payloadu: „notify” (normalna notyfikacja) i „message-only” (tylko dane). Klient rozróżnia typ i decyduje o wyświetleniu alertu systemowego.
  • W Android + natywne FCM: analogicznie można przekazać flagę silent w polu data. Aplikacja w foreground/background może wtedy zapisać wiadomość bez generowania notyfikacji systemowej.

Wniosek praktyczny: checkbox w GM.html jest prosty do wdrożenia i zgodny z obecną architekturą (GM -> trigger -> Worker -> odbiorcy). Wymaga jedynie dodania 1 pola w formularzu GM, 1 pola w żądaniu /api/push/trigger oraz warunku w logice Service Workera / FCM po stronie odbiorcy.

const SW_VERSION = "wg-pwa-v3";
const APP_SHELL_CACHE = `${SW_VERSION}-shell`;
const APP_SHELL_ASSETS = [
  "./",
  "./Main/index.html",
  "./manifest.webmanifest",
  "./IkonaGlowna.png",
  "./IkonaPowiadomien.png"
];

self.addEventListener("install", (event) => {
  event.waitUntil(
    caches.open(APP_SHELL_CACHE).then((cache) => cache.addAll(APP_SHELL_ASSETS)).catch(() => Promise.resolve())
  );
  self.skipWaiting();
});

self.addEventListener("activate", (event) => {
  event.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(keys.filter((key) => key !== APP_SHELL_CACHE).map((key) => caches.delete(key)))
    )
  );
  self.clients.claim();
});

self.addEventListener("fetch", (event) => {
  if (event.request.method !== "GET") return;

  const isNavigationRequest = event.request.mode === "navigate";

  event.respondWith(
    fetch(event.request)
      .catch(() => caches.match(event.request, { ignoreSearch: true }))
      .then((response) => {
        if (response) return response;

        if (isNavigationRequest) {
          return new Response(
            "Aplikacja wymaga połączenia z internetem. Sprawdź sieć i odśwież stronę.",
            {
              status: 503,
              headers: { "Content-Type": "text/plain; charset=utf-8" }
            }
          );
        }

        return Response.error();
      })
  );
});

self.addEventListener("push", (event) => {
  let payload = {};
  try {
    payload = event.data ? event.data.json() : {};
  } catch {
    payload = { body: event.data ? event.data.text() : "" };
  }

  const title = payload.title || "Infoczytnik";
  const body = payload.body || "+++ INCOMING DATA-TRANSMISSION +++";
  const icon = payload.icon || "/IkonaPowiadomien.png";
  const badge = payload.badge || "/IkonaPowiadomien.png";
  const url = payload.url || "/Infoczytnik/Infoczytnik.html";

  event.waitUntil(
    self.registration.showNotification(title, {
      body,
      icon,
      badge,
      tag: payload.tag || "infoczytnik-new-message",
      renotify: true,
      data: { url }
    })
  );
});

self.addEventListener("notificationclick", (event) => {
  event.notification.close();
  const targetPath = event.notification.data?.url || "/Infoczytnik/Infoczytnik.html";
  const targetUrl = new URL(targetPath, self.location.origin).href;

  event.waitUntil(
    clients.matchAll({ type: "window", includeUncontrolled: true }).then((clientList) => {
      for (const client of clientList) {
        if (client.url === targetUrl && "focus" in client) {
          return client.focus();
        }
      }

      for (const client of clientList) {
        if (client.url.includes("/Infoczytnik/") && "focus" in client) {
          return client.focus();
        }
      }

      if (clients.openWindow) {
        return clients.openWindow(targetUrl);
      }

      return Promise.resolve();
    })
  );
});

    function urlBase64ToUint8Array(base64String) {
      const padding = "=".repeat((4 - (base64String.length % 4)) % 4);

  function resolveLocalAudioUrl(rawUrl, fallbackUrl){
    if(typeof rawUrl !== "string") return fallbackUrl;

    const candidate = rawUrl.trim();
    if(!candidate) return fallbackUrl;

    // Dopuszczamy wyłącznie lokalne ścieżki (repo) — bez zewnętrznych hostów.
    if(/^assets\/audio\//i.test(candidate)) return candidate;
    if(/^\.\/?assets\/audio\//i.test(candidate)) return candidate;

    // Opcjonalnie: absolutny URL, ale tylko do bieżącego origin i ścieżki assets/audio.
    try{
      const parsed = new URL(candidate, window.location.href);
      if(parsed.origin === window.location.origin && /^\/?.*assets\/audio\//i.test(parsed.pathname)){
        return parsed.pathname + parsed.search + parsed.hash;
      }
    }catch(e){
      // Ignorujemy błędny URL i przechodzimy do fallbacku.
    }

    console.warn("[Infoczytnik] Odrzucono zewnętrzny lub nieprawidłowy URL audio:", candidate);
    return fallbackUrl;

        DEFAULT_MSG_URL
      );
      window.__dsPlayUrlOnce(msgUrl);
    }

    function drawUniqueRandomLines(source, count){
      const arr = [...source];
      for(let i=arr.length-1; i>0; i--){
        const j = Math.floor(Math.random() * (i + 1));
        [arr[i], arr[j]] = [arr[j], arr[i]];
      }
      return arr.slice(0, Math.min(count, arr.length));
