3. Docelowy przepływ pracy administratora

  1. Administrator trzyma plik Repozytorium.xlsx lokalnie lub w prywatnym miejscu. Plik nie trafia do publicznego repo.
  2. Administrator otwiera moduł DataVault w trybie admina.
  3. Administrator klika przycisk typu Wybierz Repozytorium.xlsx.
  4. Przeglądarka pokazuje okno wyboru pliku z dysku lokalnego.
  5. Administrator wskazuje lokalny plik Repozytorium.xlsx.
  6. Aplikacja przetwarza XLSX w przeglądarce przy użyciu obecnego mechanizmu generowania.
  7. Powstaje plik data.json w tym samym formacie logicznym, co dotychczas.
  8. Administrator uploaduje wygenerowany data.json do Firebase Realtime Database pod ścieżkę /datavault/live.
  9. DataVault i GeneratorNPC pobierają aktualne dane z Firebase po autoryzacji hasłem.
Ważne: Repozytorium.xlsx nigdy nie musi być pobierany przez aplikację z publicznego URL-a. Przeglądarka ma odczytać plik wyłącznie po świadomym wskazaniu go przez administratora.

4. Jak ma działać DataVault po zmianie

Tryb zwykłego użytkownika / gracza

  1. Użytkownik otwiera DataVault przez przeglądarkę lub PWA.
  2. Aplikacja sprawdza, czy istnieje aktywna sesja Firebase Auth.
  3. Jeżeli sesja istnieje, aplikacja pobiera dane z /datavault/live.
  4. Jeżeli sesji nie ma, aplikacja pokazuje ekran logowania z jednym polem hasła.
  5. Po poprawnym haśle aplikacja loguje technicznego użytkownika i pobiera dane.
  6. Dane są ładowane do dotychczasowej zmiennej/struktury roboczej DataVault, tak aby reszta logiki tabel działała jak wcześniej.

Tryb admina

  1. Admin wchodzi w DataVault w trybie admina.
  2. Przycisk Generuj data.json nie pobiera już Repozytorium.xlsx z folderu aplikacji.
  3. Przycisk otwiera wybór lokalnego pliku Repozytorium.xlsx.
  4. Po wskazaniu pliku aplikacja wykonuje ten sam proces parsowania/generowania, który istnieje obecnie.
  5. Wynikiem jest lokalnie wygenerowany data.json.
  6. Na etapie ręcznym admin importuje ten JSON w Firebase Realtime Database do ścieżki /datavault/live.

Zmiana komunikatu w UI

Obecny komunikat w DataVault mówi, że Repozytorium.xlsx musi istnieć obok index.html. Po zmianie powinien mówić:

W trybie admina wybierz lokalny plik Repozytorium.xlsx.
Aplikacja wygeneruje data.json lokalnie w przeglądarce.
Następnie wgraj wygenerowany data.json do Firebase Realtime Database pod /datavault/live.

5. Jak ma działać GeneratorNPC po zmianie

GeneratorNPC ma przestać wskazywać publiczny plik ../DataVault/data.json. Zamiast tego powinien korzystać z tego samego mechanizmu ładowania danych z Firebase, co DataVault.

Docelowe zachowanie

  1. Użytkownik otwiera GeneratorNPC.
  2. Generator sprawdza sesję Firebase Auth.
  3. Jeżeli sesja istnieje, pobiera /datavault/live.
  4. Jeżeli sesji nie ma, pokazuje ekran hasła.
  5. Po zalogowaniu pobiera dane i wypełnia listy: Bestiariusz, Bronie, Pancerze, Augumentacje, Ekwipunek, Talenty, Psionika, Modlitwy itd.
  6. Źródło danych w panelu UI powinno pokazywać opis prywatnej bazy, nie link do publicznego pliku.

Element do usunięcia / zastąpienia w GeneratorNPC

const DATA_URL = "../DataVault/data.json";

Po zmianie GeneratorNPC nie powinien wykonywać fetch(DATA_URL) dla lokalnego publicznego pliku. Ma używać wspólnego loadera Firebase.

6. Hasło, sesja, PWA i brak ciągłego pytania o hasło

Docelowo użytkownik/admin nie powinien wpisywać hasła przy każdym przełączeniu zakładki ani przy przejściu między modułami. Hasło powinno być wymagane tylko wtedy, gdy nie ma aktywnej sesji.

Docelowy model sesji

  • Aplikacja używa Firebase Authentication z trwałością sesji typu browserLocalPersistence.
  • Sesja jest zapisywana lokalnie w przeglądarce/PWA.
  • Po odświeżeniu strony aplikacja sprawdza onAuthStateChanged albo równoważny mechanizm oczekiwania na stan Auth.
  • Jeżeli użytkownik jest nadal zalogowany, aplikacja od razu ładuje dane z Firebase.
  • Jeżeli użytkownik nie jest zalogowany albo sesja wygasła, pojawia się ekran hasła.
  • Przełączanie zakładek w DataVault nie wywołuje ponownego logowania.
  • Przejście z DataVault do GeneratorNPC nie powinno wywoływać ponownego logowania, jeśli działa w tej samej domenie/originie i korzysta z tej samej konfiguracji Firebase.

Wymagany ekran dostępu

Zamiast pola e-mail + hasło użytkownik może widzieć tylko jedno pole: 

Hasło dostępu

Techniczny e-mail użytkownika Firebase może być zapisany w konfiguracji aplikacji jako identyfikator konta dostępowego. To nie jest sekret. Sekretem jest hasło i nie wolno go zapisywać w kodzie.

Opcjonalny przycisk

Warto dodać przycisk: 

Wyloguj / Zablokuj dane

Przycisk wykonuje wylogowanie z Firebase Auth i czyści dane z pamięci aplikacji.

7. Zmiany wymagane w kodzie aplikacji

7.1. Zmiany wspólne dla DataVault i GeneratorNPC

  1. Dodać wspólny moduł/plik odpowiedzialny za Firebase Auth i Realtime Database, np. shared/firebase-data-loader.js.
  2. Dodać konfigurację Firebase z databaseURL: https://wh40k-data-slate-default-rtdb.europe-west1.firebasedatabase.app.
  3. Dodać importy Firebase SDK dla firebase-app, firebase-auth i firebase-database.
  4. Ustawić trwałość sesji Auth na lokalną, aby PWA/przeglądarka nie pytała stale o hasło.
  5. Dodać funkcję logowania technicznego użytkownika przez signInWithEmailAndPassword(technicalEmail, passwordFromInput).
  6. Dodać funkcję pobierania danych z Realtime Database: ścieżka datavault/live.
  7. Dodać funkcję getCurrentData() / loadDataVaultData(), którą oba moduły będą wywoływać zamiast fetch("data.json").
  8. Dodać obsługę błędu Permission denied jako komunikatu: „Brak dostępu. Podaj hasło ponownie albo sprawdź konfigurację.”
  9. Dodać ekran blokady danych, widoczny tylko wtedy, gdy nie ma sesji.
  10. Nie zapisywać hasła w localStorage, sessionStorage, repo ani w plikach konfiguracyjnych.

7.2. Zmiany w DataVault

  1. Zastąpić ładowanie publicznego data.json pobraniem danych z Firebase Realtime Database.
  2. Po pobraniu danych z Firebase przypisać je do obecnej struktury roboczej DB, tak aby istniejące renderowanie tabel, filtrowanie, sortowanie i porównywanie działały dalej.
  3. Zachować oczekiwany format danych: { sheets: ..., _meta: ... }, czyli taki sam jak w dotychczasowym data.json.
  4. Zaktualizować statusy tekstowe: „Ładowanie data.json...” na coś w rodzaju „Ładowanie danych z prywatnej bazy...”.
  5. Zmienić przycisk/akcję Generuj data.json: zamiast pobierać Repozytorium.xlsx z folderu, ma otwierać wybór pliku lokalnego.
  6. Dodać ukryty lub widoczny input pliku: <input type="file" accept=".xlsx,.xlsm,.xls" />.
  7. Po wyborze pliku przekazać jego ArrayBuffer do istniejącego parsera XLSX.
  8. Nie zmieniać logiki transformacji XLSX → JSON, o ile nie będzie konieczności wynikającej z wejścia pliku.
  9. Po wygenerowaniu data.json umożliwić pobranie pliku przez admina.
  10. Na tym etapie nie dawać aplikacji prawa zapisu do Firebase, ponieważ reguły mają ".write": false.
  11. W komunikacie admina dodać instrukcję: „Wgraj wygenerowany data.json do Firebase Realtime Database pod /datavault/live.”

7.3. Zmiany w GeneratorNPC

  1. Usunąć zależność od const DATA_URL = "../DataVault/data.json";.
  2. Usunąć lub zmienić widoczny link do ../DataVault/data.json w panelu „Źródło danych”.
  3. Zastąpić fetch(DATA_URL) wywołaniem wspólnego loadera Firebase.
  4. Po pobraniu danych utrzymać obecną logikę pobierania arkuszy z obiektu danych, np. Bestiariusz, Bronie, Pancerze itd.
  5. GeneratorNPC ma używać tej samej sesji Firebase Auth, co DataVault.
  6. Jeżeli dane nie są dostępne, GeneratorNPC ma pokazać ekran hasła albo komunikat o braku dostępu, a nie link do publicznego pliku.
  7. Ulubione w Firestore są osobnym mechanizmem i nie powinny zostać przypadkowo pomieszane z bazą danych DataVault.

7.4. Zmiany w PWA / service worker

  1. Upewnić się, że service worker nie cache’uje DataVault/data.json.
  2. Nie dodawać data.json do listy zasobów App Shell.
  3. Nie cache’ować prywatnej odpowiedzi z Realtime Database przez service worker jako publicznego zasobu.
  4. App shell może działać offline, ale dane DataVault powinny być pobierane po autoryzacji.
  5. Jeżeli kiedyś zostanie dodany tryb offline danych, musi to być świadoma decyzja: dane będą zapisane lokalnie na zalogowanym urządzeniu.

7.5. Zmiany w repozytorium

  1. Usunąć DataVault/data.json z repozytorium, jeśli istnieje.
  2. Usunąć DataVault/Repozytorium.xlsx z repozytorium, jeśli istnieje.
  3. Dodać do .gitignore wpisy blokujące przypadkowe wrzucenie danych: 
    Repozytorium.xlsx
data.json
DataVault/Repozytorium.xlsx
DataVault/data.json
**/Repozytorium.xlsx
**/data.json
  4. Sprawdzić historię repo. Jeśli dane były wcześniej commitowane publicznie, samo usunięcie bieżących plików nie usuwa ich z historii.
  5. Nie umieszczać hasła grupowego w żadnym commicie.

8. Zasady bezpieczeństwa i czego nie robić

Nie wystarczy ukryć linku. Jeśli data.json leży pod publicznym URL-em, to dane są publiczne, nawet jeśli aplikacja nie pokazuje bezpośredniego linku.

Nie robić

  • Nie trzymać prawdziwego Repozytorium.xlsx w publicznym repo.
  • Nie trzymać prawdziwego data.json w publicznym repo.
  • Nie trzymać prawdziwego data.json w GitHub Pages.
  • Nie używać publicznego Google Drive „każdy z linkiem”.
  • Nie zaszywać hasła w JavaScripcie.
  • Nie zaszywać klucza deszyfrującego w JavaScripcie.
  • Nie nadawać wspólnemu użytkownikowi prawa zapisu do bazy, jeśli gracze znają hasło.

Robić

  • Trzymać XLSX lokalnie/prywatnie.
  • Generować JSON lokalnie po wskazaniu pliku przez admina.
  • Wgrywać JSON do prywatnej ścieżki /datavault/live.
  • Używać Firebase Auth jako bramki dostępu.
  • Utrzymać reguły Realtime Database tak, aby anonimowy dostęp zwracał Permission denied.
  • Trzymać hasło poza repo i zmieniać je, jeśli opuści zamkniętą grupę.
